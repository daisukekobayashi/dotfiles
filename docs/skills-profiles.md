# Skills Profile Design

This document records the planned skill management model for this dotfiles repository.

## Goals

- Keep globally visible skills small and focused.
- Use project-scoped skills for repository-specific work.
- Keep frequently used skill groups curated in dotfiles so each repository does not need to rediscover them.
- Prefer official `skills` CLI behavior for third-party skills whenever possible.
- Keep custom local skills owned by dotfiles.

## Scopes

### Project Scope

Project scope is the normal mode for repository work.

Running from any directory inside a Git repository should resolve the repository root with `git rev-parse --show-toplevel`.

Expected command shape:

```sh
~/.dotfiles/setup.sh skills --scope project --profile office
~/.dotfiles/setup.sh skills --scope project --profile base,office --agent codex
```

Project scope requires an explicit `--profile`. If it is omitted, setup should fail rather than installing an implicit default.

External skills are installed by the official `skills` CLI from the repository root. The CLI owns generated external skill files and the repository root `skills-lock.json`.

Custom local skills are not written into `skills-lock.json`. They are linked from dotfiles into each selected agent's project skill directory.

Project output:

```text
repo/
  skills-lock.json          # Generated by skills CLI for external skills
  .agents/
    skills/                 # Codex project skills
    skills-profile.json     # Dotfiles-specific profile metadata
  .claude/
    skills/                 # Claude Code project skills
```

Do not automatically edit `.gitignore`. Print a reminder that `.agents/skills/` and `.claude/skills/` are generated/link targets and can usually be ignored.

### User Scope

User scope is a special dotfiles bootstrap mode, not the default model for normal repositories.

`./setup.sh skills` should remain a user-scope command and should behave like:

```sh
./setup.sh skills --scope user --profile base --agent codex --agent claude-code
```

User scope should build one dotfiles-managed user skill view and link the user agent config directories to it:

```text
~/.dotfiles/.agents/user/skills/
~/.dotfiles/.agents/user/skills-profile.json
~/.agents/skills  -> ~/.dotfiles/.agents/user/skills/
~/.claude/skills  -> ~/.dotfiles/.agents/user/skills/
```

When user profiles are changed, rebuild this single user skill view instead of keeping separate view directories per profile combination. The selected profiles are recorded in `skills-profile.json`.

For Codex user skills, keep using `$HOME/.agents/skills` because that is the path documented by OpenAI Codex skills documentation. Do not rely on the `skills` CLI global Codex path if it differs.

## Dotfiles Layout

Target layout:

```text
skills/
  local/
    github-pr/
    github-issue-worktree/
    ...
  profiles/
    base.json
    office.json
    azure.json
    frontend.json
    browser.json
    data.json
    research.json
    docs.json
```

Custom skills live under `skills/local/<name>`.

Do not add a separate external skill pool/catalog file. Profiles are the source of truth for curated external skill groups.

The root `skills-lock.json` should stop acting as a global all-skills catalog after this migration. If the dotfiles repository itself needs project-scoped skills, root `skills-lock.json` can be generated for this repository like any other project.

## Profile Format

Profiles are JSON so setup can process them with Node without adding a parser dependency.

Example:

```json
{
  "description": "Office document skills",
  "includes": ["base"],
  "external": [
    {
      "source": "anthropics/skills",
      "skills": ["docx", "pdf", "pptx", "xlsx"]
    }
  ],
  "local": []
}
```

Rules:

- `includes` contains other profile names.
- `external[].source` is an `owner/repo` source passed to `npx skills add`.
- `external[].skills` contains skill names from that source.
- `local[]` contains names under `skills/local/`.
- Includes are expanded as a union.
- Duplicate skills are ignored.
- Cyclic includes are errors.
- Missing profiles, missing local skills, and malformed external entries are errors.

## Initial Profiles

### `base`

Common workflow skills used broadly across repositories.

Include:

- `find-skills`
- `using-superpowers`
- `brainstorming`
- `grill-me`
- planning, debugging, TDD, verification, and code review workflow skills
- custom GitHub workflow skills from `skills/local/`
- repository onboarding and runtime isolation local skills

### `office`

Office and document skills:

- `docx`
- `pdf`
- `pptx`
- `xlsx`

### `azure`

Azure-related skills selected from the current curated set:

- `azure-*`
- `appinsights-instrumentation`
- `azure-devops-cli`

Only include Azure skills that are intentionally useful; do not blindly install every available Azure skill.

### `frontend`

Frontend, UI, React, Tailwind, SEO, Remotion, and web design skills.

Browser automation is intentionally not included here by default because it is useful outside frontend development.

### `browser`

Browser automation and debugging skills:

- `agent-browser`
- `browser-use`
- `chrome-devtools`
- `playwright-cli`

### `data`

Data and database skills:

- `redis-development`
- `supabase-postgres-best-practices`

### `research`

Research-specific skills:

- `read-arxiv-paper`

### `docs`

Documentation lookup and library documentation skills:

- `context7-cli`

## CLI Behavior

Use Bash as the public entry point and a small Node helper for profile resolution and metadata.

Suggested helper:

```text
setup/skills-profile.js
```

Bash responsibilities:

- Parse setup flags.
- Resolve setup paths.
- Call `npx skills add` for external skills.
- Wire final user-scope links.

Node helper responsibilities:

- Load and validate profile JSON.
- Expand `includes`.
- Merge external skills by source.
- Validate local skill names.
- Write `.agents/skills-profile.json`.
- Produce a stable install plan for Bash.

Profile add/remove helper commands are intentionally out of scope for the first implementation. Edit profile JSON by hand and use `./setup.sh skills profile validate`.

## External Skills

For project scope, use official CLI installation:

```sh
npx skills add <owner/repo> \
  --skill <name> \
  --agent codex \
  --agent claude-code \
  --copy \
  --yes
```

Run this from the repository root and let the CLI manage `skills-lock.json` and agent-specific project output paths.

If an existing `skills-lock.json` will be replaced, create a backup first. Dirty Git state should produce a warning, not block execution.

## Local Skills

Local skills live under `skills/local/<name>` and are dotfiles-owned.

Project scope should symlink local skills into selected agent project skill directories:

```text
repo/.agents/skills/<name> -> ~/.dotfiles/skills/local/<name>
repo/.claude/skills/<name> -> ~/.dotfiles/skills/local/<name>
```

User scope should also symlink local skills into the generated user profile view.

If a target path already exists and is not a dotfiles-managed symlink, fail instead of overwriting it.

## Agents

Keep profile selection separate from agent selection.

Default agents:

```text
codex
claude-code
```

Examples:

```sh
~/.dotfiles/setup.sh skills --scope project --profile office
~/.dotfiles/setup.sh skills --scope project --profile office --agent codex
```

## Metadata

Write dotfiles-specific project metadata to:

```text
repo/.agents/skills-profile.json
```

This metadata should record:

- requested profiles
- expanded profiles
- selected agents
- external install plan
- local skill links
- generator name/version if useful

Do not put dotfiles-specific metadata into `skills-lock.json`.
